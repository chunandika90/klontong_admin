1. Pilihan Stack & Database

Mengapa PostgreSQL?

  • Skema relasional — products punya FK ke categories & users. Natural untuk SQL.
  • Query kompleks — filtering, sorting, pagination lebih mudah dioptimasi dengan index SQL.
  • Constraint database-level — SKU UNIQUE, NOT NULL lebih robust.
  • Soft delete — DeleteDateColumn TypeORM berjalan mulus.
  • Ekosistem mature — pg_trgm, partitioning, read replicas untuk scaling.

MongoDB cocok jika skema sering berubah atau data sangat denormalisasi — tidak relevan di kasus ini.

Stack Teknologi

  • Frontend: Nuxt 3 (SPA mode, ssr: false)
  • State: Pinia + localStorage persistence
  • Styling: Tailwind CSS
  • Backend: NestJS + TypeORM
  • Database: PostgreSQL 15
  • Auth: JWT (7d expiry) + bcrypt
  • Docs: Swagger / OpenAPI
  • Deploy: Docker Compose
Kenapa SSR dimatikan? Klontong Admin adalah internal admin tool — tidak butuh SEO atau initial load optimization. SPA mode menghilangkan kompleksitas SSR hydration yang menyebabkan bug context di production build.

2. Struktur Backend (NestJS)

src/
├── auth/               # Feature module: register, login, JWT
│   ├── dto/            # RegisterDto, LoginDto (class-validator)
│   ├── entities/       # User entity (bcrypt hash on insert)
│   ├── guards/         # JwtAuthGuard — applied per controller
│   └── strategies/     # PassportJS JWT strategy
├── products/           # Feature module: CRUD + list + stats
│   ├── dto/            # CreateProductDto, UpdateProductDto, QueryProductDto
│   ├── entities/       # Product entity (soft-delete via DeleteDateColumn)
│   ├── products.service.ts    # Business logic + transaction
│   └── products.controller.ts
├── categories/         # Feature module: category management
├── database/
│   ├── data-source.ts  # TypeORM DataSource untuk CLI
│   └── seeds/          # Seeder: 8 kategori + ~100 produk + 2 user
├── common/
│   ├── filters/        # Global exception filter
│   └── interceptors/   # Timeout interceptor (30s)
├── health.controller.ts  # GET /health untuk load balancer
└── app.module.ts       # Root: TypeORM async config, feature modules

Prinsip Utama

  • Setiap domain punya module sendiri — tidak ada circular dependency
  • Service berisi seluruh business logic; controller hanya routing + guard
  • ValidationPipe global dengan whitelist: true — payload tidak dikenal otomatis dibuang
  • Semua write operation pakai database transaction untuk atomicity
  • Response list selalu wrap { data, meta } untuk konsistensi pagination

Pola State Management

useAuthStore

Sumber kebenaran untuk auth. Token disimpan di localStorage, di-load saat middleware berjalan. Logout clear semua state.

useProductsStore

Menyimpan query state (filter, sort, page) sehingga saat navigasi kembali ke list, state tidak reset.

4. API Contract

Semua endpoint berawalan /api. Response selalu JSON. Auth via Bearer JWT di header Authorization.

MethodPathAuthDeskripsi
POST/auth/register—Register user baru
POST/auth/login—Login, dapat JWT token
GET/auth/meJWTInfo user yang login
GET/productsJWTList produk (pagination, search, filter, sort)
GET/products/statsJWTDashboard stats + per-kategori count
GET/products/:idJWTDetail produk
POST/productsJWTBuat produk baru
PATCH/products/:idJWTUpdate produk
DELETE/products/:idJWTSoft-delete (arsip) produk
PATCH/products/:id/restoreJWTRestore produk yang diarsip
GET/categoriesJWTList semua kategori
GET/health—Health check

Pagination Response

{
  "data": [ ...products ],
  "meta": { "total": 100, "page": 1, "limit": 10, "totalPages": 10 }
}

Error Response

{ "statusCode": 400, "message": ["name should not be empty"], "error": "Bad Request" }
Swagger UI tersedia di http://localhost:4001/api/docs untuk eksplorasi interaktif semua endpoint.

6. Skalabilitas: 100 → 100.000 Produk

Database

  • Index komposit: (category_id, is_active, deleted_at)
  • GIN index pg_trgm pada name dan sku untuk LIKE search
  • Table partitioning berdasarkan category_id
  • Read replicas untuk query berat

Backend

  • Ganti LIKE dengan PostgreSQL full-text search (tsvector)
  • Redis caching untuk list produk (TTL ~30s)
  • CQRS pattern — pisahkan read/write handler
  • TypeORM connection pool tuning (max: 20)

Frontend

  • Virtual scrolling untuk list panjang
  • CDN untuk static assets
  • Lazy load komponen berat

Infrastructure

  • Kubernetes / managed container service
  • Load balancer dengan health check
  • Horizontal scaling backend (stateless)
  • S3 + CDN untuk image hosting

7. Rencana RBAC

Saat ini User.role sudah ada (admin | staff). JWT payload sudah include role — tidak perlu DB lookup tambahan per request.

Implementasi

  1. Buat RolesGuard yang membaca decorator @Roles('admin')
  2. Apply ke endpoint sensitif: POST/DELETE products, POST categories
  3. Staff hanya bisa read + update stok
@Roles('admin')
@UseGuards(JwtAuthGuard, RolesGuard)
@Delete(':id')
remove(@Param('id') id: number) {
  return this.productsService.remove(id);
}

8. Rencana Audit Logging

CREATE TABLE audit_logs (
  id         SERIAL PRIMARY KEY,
  user_id    INT REFERENCES users(id),
  action     VARCHAR(50),   -- CREATE, UPDATE, DELETE, RESTORE
  entity     VARCHAR(50),   -- products, categories
  entity_id  INT,
  before     JSONB,         -- snapshot sebelum perubahan
  after      JSONB,         -- snapshot sesudah perubahan
  ip_address VARCHAR(45),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

Implementasi via NestJS interceptor atau TypeORM subscriber — tidak perlu modifikasi service yang sudah ada.

9. Yang Sengaja Tidak Dibangun

  • Email verification — tidak ada SMTP setup; tidak kritis untuk assignment
  • Cursor pagination — offset pagination cukup untuk 100 produk
  • Optimistic UI update — menambah kompleksitas state tanpa manfaat nyata di skala ini
  • Unit test frontend — backend unit test lebih kritis; Vitest setup butuh waktu lebih
  • CI/CD pipeline — Docker Compose cukup untuk deliverable; GitHub Actions bisa ditambah setelah

10. Production Gaps

🔒 Security

  • JWT secret wajib panjang & random
  • Rate limiting — endpoint login rentan brute-force
  • Helmet untuk HTTP security headers
  • HTTPS wajib di production
  • Input sanitization XSS pada description

⚡ Performance

  • Tidak ada caching layer (Redis)
  • LIKE %keyword% tidak pakai index
  • Image hosting masih URL eksternal
  • Connection pool belum dikonfigurasi

📊 Observability

  • Tidak ada structured logging (Winston/Pino)
  • Tidak ada request tracing / correlation ID
  • Tidak ada APM (Sentry, Datadog)
  • Error frontend tidak dilaporkan

🚀 Deployment

  • Docker Compose hanya untuk development
  • synchronize: true tidak production-safe
  • Tidak ada graceful shutdown
  • Tidak ada backup otomatis database

🧪 Testing

  • Hanya unit test ProductsService
  • Tidak ada integration test (real DB)
  • Tidak ada e2e test (Playwright/Cypress)
  • Tidak ada contract test frontend ↔ backend

🗄️ Data Migration

  • Perlu generate migration awal dari synchronize
  • TypeORM tidak auto-drop kolom
  • Butuh strategi backup PostgreSQL
  • Zero-downtime migration belum direncanakan
Semua gap di atas disadari penuh. Solusi ini dirancang sebagai "cukup untuk assignment" yang bisa dikembangkan ke production dengan penambahan layer-layer di atas secara bertahap.
